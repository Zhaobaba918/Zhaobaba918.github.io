跨域

同源策略会阻止一个域的javascript脚本和另外一个域的内容进行交互

是前端浏览器的问题，后端能接收到请求

是浏览器的保护机制，后端收到请求了也返回结果了

浏览器给拦击了

浏览器同源策略，不同源请求无法交互，避免攻击

url协议域名端口都相同是同源

前后端分离是两个应用肯定不同源

CORS：cross origin resource sharing

spring boot项目解决跨域

  1. 目标方法添加注解@CrossOrigin

浏览器禁止跨域的目的是防范CSRF攻击

CSRF攻击 Cross-site request forgery 是Web

常见攻击之一

前后端不分离的页面，黑客页面可能会在页面上伪装一些超链接，是这个超链接是A网站的，用用户的cookie在A网站做一些敏感的事

前后端分离的项目不怕因为点击页面上的链接的时候访问不会带token

为什么cookie不能够阻止csrf，token能，因为cookie是浏览器自动带上的，token要手动带上

session机制：

用户登录后，后端建立一个session并保存，将sessionid保存在cookie中返回前端，前端以后请求都带着这个cookie，session的过期信息什么的由后端保存

问题：

如果后端有多台服务器（Nginx是负载均衡器）

session只存在了一台服务器上（无法做到单点登录）

画外音：单点登录是在多应用系统中用户只需登录一次就可以访问所有服务

解决（大公司的鉴权方式）：

token：将用户id存在token中

只要server的秘钥不泄露，生成的token就是安全的，因为如果伪造token的话签名验证无法通过，可以判定token非法

在移动端原生请求是没有 cookie 之说的，而 sessionid 依赖于 cookie，sessionid 就不能用 cookie 来传了

token缺点：

  1. 太长
  2. 不安全：token太长不能放在cookie中，而是存在浏览器的local storage中，可以被js读取；token一旦生成无法让其失效，必须等到过期，及时服务端检测到一个威胁也无法让关联的token失效

所以token更适合一次性的认证命令，设置一个比较短的有效期

误解：cookie比token不安全，比如cross site request forgery

CSRF 攻击的根本原因在于对于同样域名的每个请求来说，它的 cookie 都会被自动带上，这个是浏览器的机制决定的，所以很多人据此认定 cookie 不安全。

token是存放在local storage中，存储角度也可以被js读取

所以不管是 cookie 还是 token，从存储角度来看其实都不安全，都有暴露的风险，我们所说的安全更多的是强调传输中的安全，可以用 HTTPS 协议来传输， 这样的话请求头都能被加密，也就保证了传输中的安全。

其实我们把 cookie 和 token 比较本身就不合理，一个是存储方式，一个是验证方式，正确的比较应该是 session vs token。

自己的理解：为什么token防止csrf？

因为cookie浏览器自动携带，token浏览器不自动携带，要js从local storage取出，可能这样麻烦一点
